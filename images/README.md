# Prebuilt QEMU Images

ArmOS release images are stored in this directory and may also be attached to
the corresponding GitHub release. They provide a build-free path for both
reference QEMU targets.

## ArmOS 0.7.4

Use one bundled archive:

- [ARM32 QEMU Virt](ArmOS-0.7.4-qemu-virt-arm32.tar.gz)
- [ARM64 QEMU Virt](ArmOS-0.7.4-qemu-virt-arm64.tar.gz)
- [ARM32 QEMU Virt kernel](kernel-arm32-qemu-virt.bin)
- [ARM64 QEMU Virt kernel](kernel-arm64-qemu-virt.bin)
- [SHA-256 checksums](SHA256SUMS)

Extract an archive from the repository root. It installs the matching kernel
and disk under the ignored local `build/images` directory:

```sh
tar -xzf ArmOS-0.7.4-qemu-virt-arm32.tar.gz
TARGET_ARCH=arm32 TARGET_PLATFORM=qemu-virt ./boot.sh
```

For ARM64:

```sh
tar -xzf ArmOS-0.7.4-qemu-virt-arm64.tar.gz
TARGET_ARCH=arm64 TARGET_PLATFORM=qemu-virt ./boot.sh
```

The repository also provides a prebuilt-image launcher. Put the downloaded
archive under `images/`; the script extracts it when needed and invokes
`boot.sh` directly without running any build:

```sh
./images/run-image.sh
./images/run-image.sh arm64
```

ARM32 is selected by default. `ENABLE_NET=1` or `ENABLE_GPU=1` can still be
provided explicitly when launching an image.

Only QEMU itself is required to boot these artifacts. QEMU 10.0.2 is the
reference version; host installation details remain in
[`INSTALLATION_macos.md`](../INSTALLATION_macos.md) and
[`INSTALLATION_linux.md`](../INSTALLATION_linux.md).

The committed archives are refreshed only for tagged releases. Development
builds continue to use the ignored local `build/images` tree generated by
`build.sh`.

## Building release assets

From the repository root, rebuild both complete QEMU targets and create the
two archives, the two raw QEMU kernels and `SHA256SUMS` under `images/`:

```sh
./tools/build_release_images.sh --version 0.7.4
```

The version must match `ARMOS_VERSION` in the root `Makefile`. To package
artifacts already present under `build/images` without recompiling:

```sh
./tools/build_release_images.sh --skip-build
```

Each run validates the ext2 root filesystem, scans the raw kernel and disk for
host information and credentials, verifies that the complete supported
userland is installed, and preserves the `build/images` extraction layout
expected by `boot.sh`. The required release payload includes TinyCC with its
Newlib development sysroot, the BSD tool suite, nano, ncurses, zlib, libjpeg,
libpng, libtiff and `fbview`.

The external XV source and binary remain excluded from public release assets
until their redistribution terms are settled. The supported image-decoding
and framebuffer path used by the local XV bring-up is included through the
libraries above and `fbview`; see [`XV_BRINGUP.md`](../docs/XV_BRINGUP.md).

The raw `kernel-arm32-qemu-virt.bin` and `kernel-arm64-qemu-virt.bin` files are
kept directly under `images/` so the latest release kernels can be versioned
without committing the much larger disk images. `SHA256SUMS` covers both
archives and both raw kernels.

## Release hygiene

Before publication, both root filesystems and archive manifests are checked
for:

- runtime Wi-Fi profiles and credentials;
- host usernames and absolute build paths;
- crash dumps, shell histories, and recovery notes;
- Finder metadata, AppleDouble files, and other hidden macOS artifacts.

Only the public example profile under `/etc/wifi.d` is retained. The ext2
filesystems are checked with `e2fsck`, and `SHA256SUMS` authenticates the
published archives.

The same host-information check can be run manually before packaging:

```sh
./tools/check_release_image_hygiene.sh \
    build/images/kernel-arm32-qemu-virt.bin \
    build/images/disk-arm32-qemu-virt.img \
    build/images/kernel-arm64-qemu-virt.bin \
    build/images/disk-arm64-qemu-virt.img
```
